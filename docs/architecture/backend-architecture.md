### 1.2 架构原则

| 原则 | 说明 |
|------|------|
| **高可用** | 无单点故障，所有组件多副本部署，故障自动切换 |
| **高并发** | 水平扩展，读写分离，缓存前置，异步解耦 |
| **高性能** | 链路优化，减少 RPC 调用，批量操作，资源复用 |
| **可观测** | 全链路追踪，指标监控，日志聚合，告警体系 |
| **可演进** | 服务解耦，模块化设计，支持灰度发布 |

## 三、服务拆分设计

### 3.1 微服务边界

```
┌─────────────────────────────────────────────────────────┐
│                     API Gateway                          │
│  (限流 / 鉴权 / 路由 / 日志 / 监控)                       │
└──────────────┬──────────────┬──────────────┬─────────────┘
               │              │              │
    ┌──────────▼───┐  ┌──────▼───────┐  ┌───▼──────────┐
    │  用户服务     │  │  行程服务     │  │  POI服务     │
    │  user-svc    │  │  trip-svc    │  │  poi-svc     │
    └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
           │                 │                 │
    ┌──────▼───────┐  ┌──────▼───────┐  ┌──────▼───────┐
    │  认证服务     │  │  行程管理     │  │  POI检索     │
    │  auth-svc    │  │  (独立?)      │  │  search-svc  │
    └──────────────┘  └──────────────┘  └──────────────┘
```

### 3.2 服务清单

| 服务名 | 职责 | 数据库 | 缓存 | MQ Topic |
|--------|------|--------|------|----------|
| **user-svc** | 用户注册、登录、个人信息、权限 | user_db | user:profile:* | user-events |
| **auth-svc** | JWT签发、刷新、OAuth、验证码 | user_db | token:* | auth-events |
| **trip-svc** | 行程CRUD、POI管理、排序、天数 | trip_db | trip:* , trip:list:* | trip-events |
| **poi-svc** | POI详情、列表、分类 | poi_db | poi:* , poi:list:* | poi-events |
| **search-svc** | 全文搜索、地理搜索、推荐 | ES | search:hot:* | - |
| **notify-svc** | 短信、推送、邮件通知 | - | - | notify-queue |
| **file-svc** | 文件上传、图片处理 | - | - | file-queue |

> **合并策略**：初期（用户<10万）可合并 user+auth、trip+poi，减少运维成本；后期按需拆分。

## 四、数据库架构

### 4.1 数据库选型

**主数据库：PostgreSQL**
- 优势：JSONB 字段支持、PostGIS 地理信息、ACID 事务强
- 适用：行程数据、用户数据、POI 基础数据

**缓存层：Redis Cluster**
- 数据结构：String / Hash / List / Set / ZSet
- 用途：用户会话、热点数据、排行榜、限流计数

**搜索引擎：Elasticsearch**
- 用途：POI 全文搜索、地理距离搜索、推荐排序

### 4.2 PostgreSQL 主从架构

```
              ┌─────────────┐
              │   应用层     │
              └──────┬──────┘
                     │
              ┌──────▼──────┐
              │  PgBouncer  │  连接池
              └──────┬──────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
   ┌────▼───┐   ┌───▼────┐   ┌──▼───────┐
   │  主库   │──▶│  从库1  │   │  从库2   │  ...
   │ (写)    │   │ (读)    │   │ (读)    │
   └─────────┘   └─────────┘   └─────────┘
        │
        ▼
   WAL 归档
   (PITR 恢复)
```

**配置要点**：
- 主库：负责所有写操作 + 强一致性读
- 从库：负责读操作（报表、列表、详情）
- 读写分离：基于 ORM 或中间件自动路由
- 连接池：PgBouncer 减少连接开销
- 备份：每日全量 + 实时 WAL 归档

### 4.3 Redis 集群架构

```
          ┌──────────────────────┐
          │   应用 (Redis Client) │
          └──────────┬───────────┘
                     │
          ┌──────────▼───────────┐
          │   Redis Cluster      │
          │  (6 节点，3主3从)    │
          └──────────────────────┘
          
  Master1 ── Slave1    Master2 ── Slave2    Master3 ── Slave3
    (0-5460)            (5461-10922)        (10923-16383)
```

**Key 设计规范**：

```
# 用户相关
user:profile:{userId}          → Hash
user:token:{token}             → String (TTL)

# 行程相关
trip:detail:{tripId}           → Hash
trip:list:{userId}             → ZSet (按创建时间)
trip:pois:{tripId}:{day}       → List (POI列表)

# POI 相关
poi:detail:{poiId}             → Hash
poi:list:{city}:{type}         → ZSet (按评分)
poi:hot:{city}                 → ZSet (热门TOP)

# 限流
rate:ip:{ip}                   → String (滑动窗口)
rate:user:{userId}:{api}       → String
```

**缓存策略**：
- **Cache-Aside**：先查缓存，未命中查 DB，再写入缓存
- **过期策略**：随机过期 + LRU 淘汰
- **穿透保护**：布隆过滤器 + 空值缓存（30s）
- **击穿保护**：互斥锁 + 热点数据永不过期
- **雪崩保护**：过期时间随机化（± 30%）

## 五、高并发方案

### 5.1 分层限流

```
┌───────────── 第1层 ─────────────┐
│  CDN / 边缘节点限流              │  静态资源、图片
└─────────────────────────────────┘
┌───────────── 第2层 ─────────────┐
│  Nginx 限流 (limit_req)         │  IP 级、接口级
└─────────────────────────────────┘
┌───────────── 第3层 ─────────────┐
│  API 网关限流 (APISIX)          │  全局限流、用户级限流
└─────────────────────────────────┘
┌───────────── 第4层 ─────────────┐
│  服务内限流 (令牌桶)            │  单实例限流
└─────────────────────────────────┘
```

**限流算法**：
- **IP 级**：滑动窗口（Redis ZSET）
- **用户级**：令牌桶（Redis + Lua 原子操作）
- **全局限流**：漏桶算法（网关层）

### 5.2 熔断与降级

**熔断器状态机**：
```
  关闭 ───(失败率>阈值)───>  打开
    ▲                          │
    │                          │  熔断时间窗
    │                          ▼
    └────────(半开探测)────── 半开
```

**降级策略**：

| 场景 | 降级方式 | 降级后表现 |
|------|----------|------------|
| POI 搜索不可用 | 降级为数据库模糊查询 | 搜索结果少，无相关性排序 |
| 推荐接口超时 | 返回热门 POI 列表 | 非个性化，但可用 |
| 行程保存失败 | 写入本地 + MQ 异步重试 | 用户无感知，后台重试 |
| 图片上传失败 | 降级为默认头像/占位图 | 不阻断主流程 |

### 5.3 异步解耦（消息队列）

**Kafka Topic 设计**：

| Topic | 生产者 | 消费者 | 用途 |
|-------|--------|--------|------|
| `trip-events` | trip-svc | search-svc, notify-svc | 行程变更同步ES、通知 |
| `user-events` | user-svc | trip-svc, notify-svc | 用户注册、资料变更 |
| `notify-queue` | 所有服务 | notify-svc | 短信、推送、邮件 |
| `file-queue` | file-svc | 图片处理Worker | 图片压缩、水印、缩略图 |
| `log-events` | 所有服务 | log-collector | 审计日志、行为日志 |

**消息可靠性**：
- 生产端：ack=all + 重试 + 本地消息表
- 消费端：幂等处理 + 死信队列 + 手动提交 offset
- 顺序性：相同业务 key 发到同一 partition

## 七、安全设计

### 7.1 网络安全

- **HTTPS 全站加密**：TLS 1.3，自动续期证书
- **WAF 防护**：SQL注入、XSS、CSRF、CC攻击
- **IP 白名单**：管理后台、内部服务
- **DDoS 防护**：云服务商高防 + 流量清洗

### 7.2 应用安全

- **认证鉴权**：JWT + Refresh Token + 黑名单机制
- **接口签名**：防止请求篡改（关键接口）
- **数据脱敏**：手机号、身份证、银行卡脱敏展示
- **密码安全**：bcrypt 哈希 + 强度校验 + 定期更换
- **SQL 注入防护**：ORM 参数化查询 + 输入校验
- **XSS 防护**：输出转义 + CSP 策略
- **CSRF 防护**：Token 校验 + Referer 校验

### 7.3 数据安全

- **数据备份**：每日全量 + 实时增量
- **异地容灾**：跨可用区部署
- **数据加密**：敏感字段加密存储（AES-256）
- **访问审计**：所有写操作记录审计日志
- **权限控制**：RBAC + 最小权限原则

## 八、性能优化

### 8.1 前端优化

- CDN 加速静态资源
- 图片懒加载 + WebP 格式
- 接口合并 + 请求防抖
- Service Worker 离线缓存
- 骨架屏 + 预加载

### 8.2 后端优化

- **数据库**：索引优化、慢查询分析、连接池、读写分离
- **缓存**：多级缓存（本地 + Redis）、热点数据预热
- **异步**：非核心逻辑异步化（MQ）
- **连接复用**：HTTP Keep-Alive、数据库连接池
- **批量操作**：批量插入、批量查询（减少 IO）

### 8.3 网络优化

- 就近接入（CDN / 边缘节点）
- HTTP/2 多路复用
- Gzip / Brotli 压缩
- DNS 预解析
